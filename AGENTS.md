# AGENTS.md — How an AI edits Baker's Recipe List

Read this before changing anything. It tells you how this site works and the
exact, enforced way to add or edit recipes. Pairs with
[`CLAUDE.md`](./CLAUDE.md) (build/deploy quick rules) and
[`docs/internal/architecture.md`](./docs/internal/architecture.md) (deep reference).

## What this repo is

A static React (Vite) single-page app that renders a personal recipe collection.
All recipe content lives in **one file**: `src/data/recipes.json` (a JSON array).
There is no database and no CMS — editing recipes means editing that JSON file,
and the live site updates only when the change is merged to `main` (GitHub Pages
deploys from `main`).

## The one rule that matters: conform to the schema, do not copy a sample

The shape of a recipe is defined once, in **`src/data/recipe.schema.json`**.
That file is the single source of truth. **Read it before adding or editing a
recipe. Do not infer the shape by looking at an existing recipe** — samples drift,
the schema does not.

The schema is enforced at build time by `scripts/validate-recipes.mjs`, which runs
as the npm `prebuild` hook. A recipe that violates the schema **fails
`npm run build` and fails CI**, so it can never reach the live site.

## Adding or editing a recipe — exact steps

1. **Open `src/data/recipe.schema.json`** and follow it field by field.
2. Edit `src/data/recipes.json`. Every record needs all 8 fields:
   `name`, `section`, `category`, `source`, `tags`, `ingredients`,
   `instructions`, `is_blank`.
   - `name` is the **globally unique** key — the app uses it for state, lookups,
     and `?recipe=` deep links. Never duplicate a name.
   - `section` must be one of the keys in `src/data/sections.js`
     (the schema's `section.enum` mirrors it; the validator fails if the two
     ever drift apart). `category` must be one of the schema's `category` enum.
   - `tags` are lowercase kebab hashtags: `#pork`, `#seasoning`, `#middle-eastern`.
   - Use unicode fractions to match the file (`½`, `¼`, `¾`), not `1/2`.
   - `source`: use `"Original"` for the author's own recipes.
3. Set `is_blank` correctly:
   - `false` = a real recipe. It **must** have at least one `type: "item"`
     ingredient and at least one instruction.
   - `true` = a placeholder shown as a "coming soon" row. Ingredients/instructions
     may be empty. Use this rather than shipping an empty recipe as if it were real.
4. **Validate before committing:** `npm run validate:recipes`
   (or just `npm run build`, which runs it first). Fix every error it reports.
5. Commit on a branch and open a PR (see below). Do not edit `dist/`.

## Two instruction conventions (both valid)

`instructions` is an array of step objects. The collection uses two shapes — the
schema and validator accept both:

- **Classic:** `{ "step": "<short title>", "detail": "<full instruction>" }`.
  Use this for new recipes. `detail` carries the real text.
- **Grouped** (used by multi-version marinades): `{ "type": "...", "step":
  "<full content>", "detail": "" }`, where `type: "section"` / `"header"` is a
  sub-heading like `"Version 1 - Food Network"` and `type: "item"` is a step.
  Here the content lives in `step` and `detail` is empty.

Rule the validator enforces: `step` is always non-empty; `detail` is required but
may be empty; `type`, when present, is `item`, `section`, or `header`.

## Workflow & guardrails (from CLAUDE.md)

- `main` is **protected** — never `git push origin main`. Branch → commit → push
  → open a PR → squash-merge once CI is green. Merging deploys to GitHub Pages.
- `npm run lint` must pass before committing.
- **Never hand-edit `dist/`** — it is generated by the build.
- Root is locked: new CSS → `src/styles/`, new component → `src/components/`,
  new util → `src/utils/`, build script → `scripts/`, planning doc → `docs/internal/`.

## Known tech debt — the `FOR REVIEW` sections

The `FOR REVIEW ...` section keys are a staging area surfaced under the app's
"For Review" tab (driven by `review: true` in `sections.js`, and referenced by
`autoTags.js` and `estimateServings.js`). They overload `section` to encode both
a category and a review status, with inconsistent delimiters
(`FOR REVIEW ---` vs `FOR REVIEW -`) and duplicate soup buckets. They are valid
for now (locked in the schema enum) but are slated for a refactor to a dedicated
`status` field. **Do not add new recipes under a `FOR REVIEW` section** unless you
are deliberately staging something for review.
